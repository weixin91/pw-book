# Feature Specification: Android 自动填充凭据索引优化

**Feature Branch**: `001-password-manager`
**Created**: 2026-05-05
**Status**: Draft
**Input**: User description: "对Android app进行类似 '2.1 ✅ 同步时全量解密' 的索引优化，针对自动填充场景。不要影响现有功能正常使用，充分考虑索引更新的场景。"

## Clarifications

### Session 2026-05-05

- Q: 锁定状态下接收同步事件时，索引应如何处理? → A: 锁定时记录"待重建 cipherId 集合"（增/改入集合，删除即时移除索引条目），下次解锁后仅对集合内 cipherId 增量重建对应索引条目。
- Q: 自动填充"保存"路径（SaveRequestHandler 查重）是否纳入索引优化范围? → A: 纳入；fill / Passkey Get / Save 三条路径共享同一索引并均走索引筛选。
- Q: 解锁时检测索引"一致性"的判断标准是什么? → A: 比对索引中 cipherId 集合与本地凭据 cipherId 集合是否完全相等；不相等则触发重建（仅缺失/多余的条目，配合"待重建集合"做最小增量重建）。
- Q: 索引筛选后是否必须对解密后的凭据再做"二次校验"? → A: 必须；fill / Save 路径用 UriMatcher、Passkey 路径用 PasskeyMatcher 对解密后的候选凭据再校验一次，确保最终行为与未启用索引时完全等同。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 网页/应用自动填充快速响应 (Priority: P1)

用户在浏览器或第三方 App 的登录页打开输入焦点，系统弹出 Password Book 的自动填充建议。当用户保险库中已保存数百条凭据时，系统能在不感知卡顿的时间内完成域名匹配并展示建议，避免每次填充都触发对全部凭据的解密。

**Why this priority**: 自动填充是用户与移动端最高频的交互场景，每次都全量解密会导致明显延迟、功耗增加，并直接影响用户体验。该优化是其他 Passkey 与编辑流程性能改进的基础。

**Independent Test**: 在保险库中创建 300 条不同域名的凭据，打开任意网页并触发自动填充，观察建议出现的时间是否稳定低于 500 毫秒；同时关闭索引特性后对比响应时间，验证优化有效。

**Acceptance Scenarios**:

1. **Given** 保险库已解锁且包含 300 条凭据，**When** 系统调用自动填充服务且当前页面与其中 2 条凭据匹配，**Then** 系统仅对匹配的 2 条凭据执行解密，其余凭据无需解密即可完成响应。
2. **Given** 保险库已解锁且包含 0 条凭据，**When** 系统调用自动填充服务，**Then** 系统直接返回空建议而不进行任何解密操作。
3. **Given** 保险库已锁定，**When** 系统调用自动填充服务，**Then** 系统返回"解锁 Password Book"提示，索引不被读取或更新，行为与现状一致。
4. **Given** 保险库中存在与当前页面通过 DomainAssociation 规则匹配的凭据，**When** 系统调用自动填充服务，**Then** 索引筛选阶段必须考虑域名关联规则，最终匹配结果与未启用索引时完全一致。
5. **Given** 用户在网页上提交了登录表单触发自动填充保存，**When** 系统判断该凭据是否已存在，**Then** 系统通过索引筛选候选 cipherId 集合后再仅对其解密做最终比对，不再遍历解密所有凭据。

---

### User Story 2 - Passkey 凭据快速定位 (Priority: P2)

用户在网页或应用上触发 Passkey 登录请求，系统显示可用的 Passkey 凭据列表。当用户保险库中存有大量凭据但仅少数为 Passkey 时，系统能跳过非 Passkey 凭据的解密，直接通过 rpId 索引定位候选凭据。

**Why this priority**: Passkey 流程对用户感知延迟同样敏感（系统弹窗等待时间）。当前 `findByRpId` 会遍历所有 LOGIN 类型凭据并解密匹配 rpId，是另一个全量解密热点。

**Independent Test**: 保险库中包含 200 条普通登录凭据 + 5 条 Passkey 凭据，发起任意 Passkey GetCredential 请求，验证仅对包含 Passkey 的少量凭据进行解密，且最终列出的候选 Passkey 与未启用索引时一致。

**Acceptance Scenarios**:

1. **Given** 保险库中含有 200 条非 Passkey 凭据和 5 条 Passkey 凭据，**When** Credential Provider 收到 PublicKeyCredential 请求，**Then** 系统先通过索引筛选 hasPasskey=true 的凭据，再仅对其中 rpId 匹配的条目解密。
2. **Given** 请求中携带 allowCredentials 限定列表，**When** 系统检索 Passkey 候选，**Then** 索引筛选不影响 allowCredentials 的过滤逻辑，最终结果与未启用索引时一致。
3. **Given** rpId 通过 DomainAssociation 与调用方包名关联，**When** 系统检索 Passkey 候选，**Then** 索引筛选必须能正确返回这些跨域关联的 Passkey 候选。

---

### User Story 3 - 索引随凭据变更保持一致 (Priority: P1)

用户对凭据进行任意增删改操作（手动编辑、自动填充保存、Passkey 创建/更新、远程同步、登出/重新登录）时，索引始终与最新凭据数据保持一致，不出现"索引已删除但凭据仍在"或"凭据已更新但索引仍指向旧域名"等错配情况。

**Why this priority**: 索引一旦不一致，用户会遇到自动填充建议缺失、Passkey 不可见等严重功能缺陷，违反"不影响现有功能"的硬性要求。索引一致性的优先级与自动填充功能本身相同。

**Independent Test**: 在每种凭据变更入口下分别执行一次操作，操作前后导出索引快照与凭据列表对比，确保索引中的 cipherId、域名、rpId 与凭据解密结果完全匹配；并模拟服务进程重启，验证索引可恢复。

**Acceptance Scenarios**:

1. **Given** 用户在编辑页保存或新增一条凭据，**When** 保存完成，**Then** 该凭据的索引条目立即被新增或更新，无需等待下一次同步。
2. **Given** 用户在保险库列表中删除一条凭据，**When** 删除完成，**Then** 对应索引条目立即被移除，后续自动填充不会出现该凭据。
3. **Given** 远程同步执行全量同步并清空本地凭据后重写，**When** 同步完成，**Then** 索引被完整重建，与新的凭据集合一致。
4. **Given** 远程同步执行增量同步并下发新增、更新、删除事件，**When** 同步完成，**Then** 索引按事件类型增量更新，无需对未变更凭据重新解密。
5. **Given** 自动填充服务进程被系统回收后再次唤起，**When** 下一次填充请求到达，**Then** 索引能够从持久化存储恢复，无需重新解密所有凭据。
6. **Given** 用户解锁保险库，**When** 索引中的 cipherId 集合与本地凭据 cipherId 集合不相等（包括数量相同但 ID 集合不同的情况），**Then** 系统在解锁完成后自动针对差异部分重建索引，不阻塞解锁流程。
7. **Given** 用户登出（清空本地凭据），**When** 退出完成，**Then** 索引被清空，避免遗留旧用户数据。

---

### Edge Cases

- 凭据加密 JSON 中缺少 `login.uris` 或 `passkey` 字段时，对应索引字段为空数组/null，但条目本身仍需登记以支持 hasLogin/hasPasskey 标记。
- 单条凭据解密失败（数据损坏、密钥错配）时，应跳过该条索引建立但不中断整体重建流程，并记录日志。
- 凭据 URI 含有非 HTTP/HTTPS、`androidapp://` 之外的自定义 scheme 时，索引保留原始 baseDomain 提取规则的输出（与 UriMatcher 行为一致），不引入新规则。
- 升级安装首次运行时（旧版本无索引），系统在保险库下次解锁时按需重建一次。
- 索引存储被外部清理（用户清空 App 数据、备份恢复）时，系统在下次解锁/同步时自动重建。
- 锁定状态下接收同步推送（凭据增量更新）时，系统不解密新凭据数据；新增/更新事件的 cipherId 加入"待重建集合"，删除事件即时清除对应索引条目;下次解锁后仅对集合内 cipherId 重建索引,集合处理完成后清空。

## Requirements *(mandatory)*

### Functional Requirements

#### 索引数据结构

- **FR-001**: 系统必须为每条 LOGIN 类型凭据维护一条索引条目，包含字段：cipherId、域名列表、rpId 列表、hasLogin 标记、hasPasskey 标记。
- **FR-002**: 索引条目仅存储非敏感信息，必须不包含明文密码、私钥、TOTP 种子、明文用户名、明文备注。
- **FR-003**: 域名列表必须由 `login.uris` 中每条 URI 经过现有 UriMatcher 域名提取规则后产生，与现有匹配逻辑保持一致。
- **FR-004**: rpId 列表必须取自 `passkey.rpId` 字段并统一转为小写，与现有 PasskeyMatcher 行为一致。

#### 自动填充与 Passkey 查询

- **FR-005**: 自动填充查询凭据时必须先通过索引筛选候选 cipherId 集合，再仅对该集合中的凭据执行解密。
- **FR-005a**: 自动填充保存（表单提交后查重）路径必须先通过索引按目标 URL/域名筛选候选 cipherId 集合，再仅对该集合解密以判定是否存在等价凭据；不得遍历解密全部凭据。
- **FR-006**: Passkey GetCredential 请求必须先通过 hasPasskey=true 与 rpId 索引筛选候选 cipherId 集合，再仅对该集合执行解密。
- **FR-007**: 索引筛选阶段必须支持 DomainAssociation 跨域关联与子域匹配规则，结果集合必须为现有匹配逻辑结果集合的超集（允许略宽松）。
- **FR-007a**: 系统必须对索引筛选返回的候选 cipherId 解密后的数据再用现有 `UriMatcher`（fill / Save 路径）或 `PasskeyMatcher`（Passkey 路径）做二次校验，最终参与构建自动填充建议或返回给调用方的凭据集合必须与未启用索引时完全等同。
- **FR-008**: 当索引缺失、未初始化或与本地凭据 cipherId 集合不相等时，系统必须降级为现有全量解密逻辑，确保功能不被打断。

#### 索引生命周期

- **FR-009**: 用户解锁保险库后，系统必须比对索引中的 cipherId 集合与本地凭据的 cipherId 集合；若不相等，对差异部分（多余条目移除、缺失条目重建）执行修复，并合并 Q1 中"待重建集合"内的 cipherId 一起处理；该修复在后台执行，不阻塞解锁与首次自动填充请求。
- **FR-010**: 用户在编辑页、自动填充保存、Passkey 创建/更新流程中保存或修改凭据后，系统必须立即更新该凭据对应的索引条目。
- **FR-011**: 用户删除单条凭据后，系统必须立即移除该凭据对应的索引条目。
- **FR-012**: 远程全量同步（清空并重写本地凭据）完成后，系统必须重建整个索引。
- **FR-013**: 远程增量同步下发新增、更新、删除事件后，系统必须按事件类型对索引执行 upsert 或 remove，不重新解密未变更的凭据。
- **FR-014**: 用户登出或切换账号时，系统必须清空索引存储，避免遗留旧用户数据。
- **FR-015**: 索引必须以可在自动填充服务进程重启后恢复的方式持久化，避免每次进程重建都重新解密全部凭据。

#### 安全与一致性

- **FR-016**: 索引存储必须遵守与凭据元数据（id、modifiedAt 等）相同的访问控制级别，不得放宽到无保护的明文文件。
- **FR-017**: 在保险库锁定状态下接收同步事件时，系统必须不调用 userKey、不解密任何凭据；对新增/更新事件，将 cipherId 加入"待重建集合"持久化保存；对删除事件，立即从索引中移除对应条目（仅依赖 cipherId，不读取明文）。
- **FR-018**: 单条凭据索引建立失败时，系统必须跳过该条目并记录日志，但不得中断其他凭据的索引建立。
- **FR-019**: 索引相关操作必须在非 UI 线程执行，避免阻塞自动填充回调或 UI。
- **FR-020**: 解锁完成后，若"待重建集合"非空，系统必须仅对集合内的 cipherId 触发增量重建，重建完成后清空集合；该过程与 FR-009 的整体一致性重建可合并执行。

### Key Entities *(include if data involved)*

- **CipherIndexEntry**: 单条凭据的非敏感匹配信息，由 cipherId（与凭据实体外键关系）、域名列表、rpId 列表、hasLogin、hasPasskey 字段组成。生命周期与对应 CipherEntity 一致。
- **CipherIndexStore**: 索引集合的持久化存储，按用户隔离，能够被自动填充服务进程在冷启动时直接读取。
- **CipherEntity**（既有）: 索引的数据来源；索引派生于其加密 JSON 解密后的内容。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 在保险库含 300 条凭据时，自动填充建议从触发到展示的端到端耗时较启用索引前下降至少 60%，且 95 分位耗时不超过 500 毫秒。
- **SC-002**: 单次自动填充请求平均解密次数从全量降至小于等于实际匹配数 + 1（用于二次校验的安全余量）。
- **SC-003**: 索引一致性验证（索引 cipherId 集合与本地凭据 cipherId 集合完全相等）在执行新增、更新、删除、全量同步、增量同步、登出、再次登录、锁定状态下增量同步并解锁等操作后通过率为 100%。
- **SC-004**: 自动填充服务进程被系统回收后再次启动，首次填充请求耗时较冷启动重新解密下降至少 50%。
- **SC-005**: 现有自动填充、Passkey 注册、Passkey 登录、保险库 CRUD、同步等场景的功能回归测试 100% 通过，无新增失败用例。
- **SC-006**: 在锁定状态下，无论索引是否被读取，用户可见数据不超过当前已展示信息（如域名列表），且不出现绕过锁定获取明文凭据的路径。

## Assumptions

- 索引可由解密后的非敏感字段派生而成，因此可以以"非加密但访问受控"的形式持久化，与 Edge 扩展中存放在 `chrome.storage.local` 的策略对齐。Android 端将沿用 Room 数据库（与现有 CipherEntity 同库）保存，访问权限受 Android 沙箱保护。
- 用户名哈希在自动填充主流程中暂不需要（Android 当前未通过用户名做候选去重），先不纳入索引以降低复杂度；后续如需 Edge 端的去重判断功能，再扩展该字段。
- 锁定状态下的同步事件，仅更新加密后的凭据原文与 modifiedAt；新增/更新事件的 cipherId 加入持久化的"待重建集合"，删除事件即时清除对应索引条目；下次解锁后按集合做增量重建，避免全量重建带来的性能负担，同时保持"不读取明文、不绕过锁定"。
- 现有 `UriMatcher.parseUri` 与 `getBaseDomain` 行为足够稳定，索引条目按其当前规则提取的域名结果可视为权威值；如这些规则未来更新，索引需重建一次。
- 索引重建期间，自动填充走全量解密兜底逻辑，避免因首次构建延迟导致建议缺失。
- 数据库迁移按现有 Room migration 流程处理，旧版本升级到新版本时新增索引表，由首次解锁触发数据回填，无需用户手动操作。
- 索引在登出时清空；账号切换视为登出 + 登录，因此索引会随之重建。
- 性能基线测试以 Pixel 6 / Android 14 为参考机型，凭据规模 300 条作为典型上限值。
